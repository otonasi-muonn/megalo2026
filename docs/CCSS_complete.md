# 🚀 プロジェクト概要: CCSSキメラアーキテクチャ (独立コンパイラ先行版)

## 1. コンセプト
* **一言でいうと**: Reactコードを読み込み、C言語とCSSハックへ「強制退化」させる独立した変換ツールを備えた、JSゼロUIとJS駆動ゲームのキメラ型Webアプリケーション。
* **解決したい課題**: モダンWeb開発の象徴であるReactの仮想DOMや状態管理を、あえて古典的なC言語の文字列生成とCSS擬似クラスに置き換える技術的退行を起こすこと。既存のビルド機構に依存せず、抽象構文木を直接操作する独自コンパイラを構築することで、ハッカソンにおける極限の「技術の無駄遣い」を体現します。
* **ターゲット**: Webアーキテクチャの限界や、泥臭いコンパイラ開発の価値を評価できる技術者および審査員。

## 2. 技術スタック
| 領域 | 採用技術 | 備考（背景知識と役割） |
| :--- | :--- | :--- |
| トランスパイラ (独自) | TypeScript | 単独で稼働する変換エンジン。Reactの抽象構文木を解析し、C言語の出力関数とCSS状態管理ロジックへ翻訳・解体します。 |
| フロントエンド (UI) | C言語 (WebAssembly) + CSS | WebAssembly化されたC言語がHTML文字列を生成し、CSSの擬似クラスがUIの状態管理を担当。仕様上不可避な「生成されたHTMLを画面に貼るための極小の接着用JS」のみ許容します。 |
| フロントエンド (ゲーム) | JavaScript (KAPLAY) | キャンバス要素上でのみ稼働。スワイプ操作による風力生成と物理演算パズルアクションを担当。 |
| バックエンド | Hono | データの受け渡しを担うAPIサーバー。通信結果を表現するための「動的CSS」を直接生成して返却する特殊な役割も担います。 |
| データベース | Supabase (PostgreSQL) | 作成されたステージデータやプレイ結果の永続化。 |
| インフラ | Vercel | 全体のホスティングおよび、WebAssemblyのコンパイル環境。 |

## 3. 機能一覧 (MVP)
初期リリースで実現する中核的な機能と技術的挑戦は以下の通りです。

* [ ] **独立型トランスパイル機構**: 指定したReactファイルを読み込み、抽象構文木を解体してC言語ソースとCSSファイルへ物理的に分割・出力するツールの構築。
* [ ] **C言語ランタイムによるDOM構築**: WebAssembly上でポインタを直接操作し、データ（JSON等）を手動解析してHTML文字列を動的に再構築するステートマシン。
* [ ] **JS非依存のUI状態管理**: 状態に対する非表示チェックボックスの識別子自動採番と、CSSの間接セレクタを駆使したモーダル開閉やタブ切り替えの実現。
* [ ] **画面遷移なしのフォーム通信と動的CSS上書き**: 非表示のインラインフレームを標的としたデータ送信と、サーバーから返却された絶対配置のCSSルールによるJS監視なしの画面強制更新ハック。
* [ ] **物理演算パズルアクション**: JavaScriptによって描画・制御される、スワイプ操作を用いた風力生成ゲーム。
* [ ] **自動ビルド統合 (追加要件)**: 独立ツールとして完成した変換エンジンを、最終的に標準のビルド処理へ組み込み全自動化する機構。

## 4. UI/UX・画面構成
画面の切り替えや表示の制御は、すべてCSSのハックを基盤として動作します。

* **ホーム / ダッシュボード画面**: C言語が生成した静的なHTMLの枠組みを利用し、公開済みのステージ一覧を展開します。ローディング等の状態変化はチェックボックスの切り替えで表現します。
* **ステージ制作画面**: メニューや設定UI部分はC言語とCSSで構築されます。中央のキャンバス部分でのみJavaScriptが稼働し、ギミックの配置を行います。保存処理は隠しインラインフレームへのフォーム送信で行われます。
* **プレイ画面**: 風を起こすアクションはJavaScriptによって処理されますが、リトライやポーズといった周辺操作はCSSハックの領域として完全に分断された状態で機能します。
* **結果画面**: ゲーム層からの判定や、サーバーからの保存完了通知を受け取った後、それをトリガーとして動的に生成されたCSSが画面全体を覆い尽くし、リザルト画面を描画します。

## 5. フォルダ構成の論理構造
モノレポの全体構造を活かし、「変換器そのもの」と「アプリケーション」を明確に分割して配置します。

* **パッケージ領域（システムの頭脳・エンジン）**
    * **コンパイラ層**: 独立して稼働する独自トランスパイラを配置します。入力からC言語ソースおよびCSSファイルへの純粋な変換ロジックをここで担保します。
    * **C言語ランタイム層**: ブラウザ上でメモリ管理やHTML文字列の組み立てを実行するC言語プログラム群を配置します。
    * **共有データ層**: システム全体で共有するデータ型の定義を管理します。
* **アプリケーション領域（実行・開発基盤）**
    * **フロントエンド層**: 入力元となるReactコードとゲームロジックを配置します。開発時は通常のReactとして動作しますが、コンパイラを通すことでUIロジックが解体されます。
    * **バックエンド層**: 特殊なCSS返却エンドポイントを含むAPIサーバーロジックを配置します。

## 6. 技術的制約とリスクマネジメント (設計上の重要留意事項)
本アーキテクチャという極端な技術的挑戦を破綻なく着地させるため、以下の厳密な設計制約を設けます。

* **抽象構文木変換対象の極小化（独自サブセットの定義）**
Reactの構文や状態管理の表現力は非常に柔軟ですが、そのすべてをC言語とCSSに変換する汎用コンパイラをハッカソン期間内に開発することは不可能です。そのため、「このツールで変換できるReactコードの書き方」をプロジェクト初期段階で極限まで限定し、厳密なサブセットとして定義します。これにより、コンパイラ実装の泥沼化を未然に防ぎます。
* **C言語側での文字列処理の安全性確保**
通信データの解析や動的なHTML文字列の連結をC言語で行う都合上、ポインタ操作の誤りによるメモリリークやバッファオーバーランの危険が常に伴います。これを回避するため、動的なメモリ確保を極力排除し、静的に確保された巨大な文字列バッファの枠組みに抽出した値をはめ込むだけの、安全かつ軽量な処理方式を設計の前提とします。
* **非同期的な状態不整合の防止（単方向イベントフロー）**
本システムでは「CSSハックで管理されるUI状態」「JavaScriptで稼働するゲームの内部状態」「サーバーが保持する永続化データ」という、全く異なる3つの次元で状態が並行稼働します。これらが複雑に干渉すると予期せぬ挙動を引き起こすため、状態が交わる境界（例：ゲーム終了時の結果画面への遷移など）においては、イベントフローを厳密に「一方向」に制限する疎結合な設計を適用します。
