# 🧪 CCSSリスク・検証設計

## 1. 主要リスク

## R1. 変換対象の肥大化

- **内容**: React全構文へ拡大すると実装破綻する。
- **対策**: `01_architecture-and-compiler.md` のサブセット制約を厳守し、逸脱はエラー停止。

## R2. C文字列処理の安全性

- **内容**: バッファ破壊やメモリリークが発生し得る。
- **対策**: 固定長バッファ中心設計、境界チェックマクロ、ASan付き検証を実施。

## R3. 状態同期の分裂

- **内容**: CSS状態・KAPLAY状態・DB状態が乖離する。
- **対策**: 境界イベントを契約化し、単方向フロー（Game -> API -> UI）を固定。

## R4. 動的CSSの安全性

- **内容**: 任意CSS注入によるUI破壊・情報漏えい。
- **対策**: 許可セレクタホワイトリスト、危険トークン拒否、監査ログ保存。

## 2. 検証マトリクス

| 分類 | 検証内容 | 合格条件 |
| --- | --- | --- |
| 変換器 | 許可構文/禁止構文テスト | 禁止構文は必ず失敗し、位置情報を返す |
| WASM | 初期化・レンダリング | 初回描画と再描画が規定時間内に完了 |
| CSS状態 | モーダル/タブ/ローディング遷移 | 期待state遷移のみ発生し逆遷移しない |
| API | 認証/認可/エラー整形 | code/message/hintの3要素を常に返す |
| ゲーム境界 | `game:end` 連携 | 結果画面遷移とplay_logs記録が両方成功 |

## 3. Done定義（設計フェーズ）

- `docs/ccss/` 内の全設計書が相互参照されている
- 変換器サブセット仕様と検証項目が一致している
- API/DB/状態ID命名規約が文書間で矛盾しない
- 失敗時の復旧手順が `05_build-and-deployment.md` と整合している

## 4. Done定義（実装フェーズ移行条件）

- サンプル1画面をReact -> C/CSSへ変換できる
- C/WASM UIとKAPLAYを同一画面で共存稼働できる
- `style-patch` APIで1つの状態遷移を実運用と同等に再現できる
- 監査ログから1セッションを時系列追跡できる
